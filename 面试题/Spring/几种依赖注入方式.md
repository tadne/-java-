# 几种依赖注入方式

依赖注入（DI）是 IoC 的实现：在运行时由容器把依赖对象注入目标 Bean。

## 三种方式对比

| 方式 | 实现 | 优点 | 缺点 |
|------|------|------|------|
| **构造器注入** | 有参构造 + `@Autowired` | 依赖明确、对象不可变、易单测 | 参数多时代码冗长；**构造器循环依赖无法解决** |
| **Setter 注入** | setter + `@Autowired` | 可选依赖、可重新注入 | 可能忘记调用 setter |
| **字段注入** | 字段上 `@Autowired` | 代码简短 | 难 mock、隐藏依赖、不推荐用于核心 Bean |

## Spring 官方推荐

1. **优先构造器注入**（必需依赖）
2. Setter 用于可选依赖
3. 避免字段注入（团队规范可禁止）

## 注解示例

```java
@Service
public class OrderService {
    private final OrderRepository repo;

    @Autowired  // Spring 4.3+ 单构造器可省略
    public OrderService(OrderRepository repo) {
        this.repo = repo;
    }
}
```

## 与循环依赖

| 注入方式 | 单例循环依赖 |
|----------|--------------|
| 字段 / setter | 三级缓存可解决 |
| 构造器 | ❌ 无法解决，需 @Lazy 或改设计 |

## 面试要点

- 三种方式都能实现 DI，推荐构造器注入。
- 构造器注入无法解决构造器循环依赖。

## 面试一句话

> Spring 支持构造器、setter、字段三种注入，推荐构造器注入保证依赖不可变，字段注入仅适合简单场景。

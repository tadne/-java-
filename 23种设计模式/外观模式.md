# 外观模式

## 思想

为子系统中的一组接口提供一个**统一的高层入口**，隐藏内部复杂调用，降低客户端与子系统的耦合。

## 结构

- **Facade**：外观类，封装对多个子系统类的调用
- **Subsystem Classes**：子系统，Facade 不增加新逻辑，只做编排

## 代码示例

```java
class CPU { void start() {} }
class Memory { void load() {} }
class Disk { void read() {} }

class ComputerFacade {
    private CPU cpu = new CPU();
    private Memory mem = new Memory();
    private Disk disk = new Disk();
    public void boot() {
        cpu.start();
        mem.load();
        disk.read();
    }
}
// 客户端只调 facade.boot()，不关心内部
```

## 典型应用

- SLF4J 对多种日志实现的门面
- Spring `JdbcTemplate` 封装 JDBC 细节
- 三层架构中 Service 对 DAO 的编排（广义上类似）

## 与适配器、代理

| | 外观 | 适配器 | 代理 |
|---|------|--------|------|
| 目的 | **简化**子系统 | **转换**接口 | **控制**访问 |
| 对象数 | 通常协调**多个** | 通常**一个** | 通常**一个** |

## 面试一句话

> 外观为复杂子系统提供单一入口，客户端只依赖 Facade；不是新功能，是简化调用。

# 面向对象设计原则

面向对象设计中常用 **SOLID + 组合复用 + 迪米特** 等原则，用于降低耦合、提高可维护性。设计模式是这些原则在具体场景下的**落地套路**。

## SOLID

| 原则 | 英文 | 要点 | 反例 |
|------|------|------|------|
| 单一职责 | SRP | 一个类只因**一种原因**变化 | 一个类又写 SQL 又发邮件 |
| 开闭 | OCP | **扩展开放**，修改关闭 | 每加支付方式改一大段 if |
| 里氏替换 | LSP | 子类可替换父类且程序正确 | 子类抛父类没有的异常 |
| 依赖倒置 | DIP | 依赖**抽象**，不依赖具体 | Service 直接 `new DaoImpl` |
| 接口隔离 | ISP | 接口精简，不强迫实现无用方法 | 胖接口 20 个方法只用 2 个 |

## 其他重要原则

| 原则 | 要点 |
|------|------|
| **组合/聚合复用** | 优先「有一个」组合，而非盲目继承 |
| **迪米特法则** | 只与直接朋友通信，少跨层调用（最少知识） |

## 原则 → 模式 对照（帮助记忆）

| 原则 | 常见模式 |
|------|----------|
| OCP | 策略、工厂方法、模板方法 |
| DIP | 工厂、依赖注入（Spring） |
| SRP | 门面（拆分职责到 Facade 背后子系统） |
| 组合复用 | 装饰、策略、桥接 |

## 示例：开闭原则

```java
// 不好：每加策略改 if
if ("ali".equals(type)) { /* ... */ }

// 更好：策略模式 + Map
Map<String, PayStrategy> strategies = ...;
strategies.get(type).pay(amount);
```

## 示例：依赖倒置

```java
// 不好
class OrderService {
    private MySqlOrderDao dao = new MySqlOrderDao();
}

// 更好：依赖接口，构造器注入
class OrderService {
    private final OrderDao dao;
    OrderService(OrderDao dao) { this.dao = dao; }
}
```

## 面试一句话

> SOLID 是设计原则，23 种模式是原则的实践；最常考 SRP、OCP、DIP 及与 Spring 依赖注入的关系。

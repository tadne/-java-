# Minor GC 和 Full GC

## 概念

| 类型 | 范围 | 触发频率 | 停顿 |
|------|------|----------|------|
| **Minor GC** | 新生代（Eden + Survivor） | 较频繁 | 通常较短 |
| **Major GC** | 老年代 | 较少 | 较长 |
| **Full GC** | 整堆 + 方法区/元空间等（收集器定义不同） | 应尽量少 | 最长，STW |

> 不同收集器对 “Major” / “Full” 的定义略有差异，面试可说：**Full GC 一般整堆回收，STW 最明显**。

## Minor GC 触发

1. **Eden 满**：新对象无法分配
2. 存活对象复制到 Survivor，年龄达标晋升老年代
3. **Survivor 放不下** 时提前晋升

特点：复制算法，存活对象少时很快。

## Full GC 常见触发（更新版）

| 触发原因 | 说明 |
|----------|------|
| 老年代空间不足 | 晋升失败、大对象、老年代碎片 |
| 元空间不足 | 类加载过多，`-XX:MetaspaceSize` |
| `System.gc()` | 建议 GC，不保证执行，生产慎用 |
| CMS 失败 | `concurrent mode failure`（CMS 已逐步淘汰） |
| G1 并发失败 |  evacuation 失败等 |
| 分配担保失败 | Minor GC 前老年代连续空间不足（相关参数与收集器有关） |

> **已修正**：元空间不足应调 `MetaspaceSize` / 排查类加载，**不应**笼统写“用 CMS 解决元空间问题”。CMS 主要管堆，且 JDK 14+ 已移除。

## 现代收集器建议（了解）

| 场景 | 推荐 |
|------|------|
| JDK 8 默认 | Parallel GC |
| 低延迟服务 | **G1**（JDK 9+ 默认）、ZGC、Shenandoah |
| 大堆低停顿 | ZGC / Shenandoah |

## 调优方向（原则）

- 减少 Full GC 频率：合理堆大小、避免大对象、减少无效类加载
- 不要依赖 `System.gc()`
- 用 `jstat`、`GC 日志`、APM 看 GC 原因，再调参数

## 面试一句话

> Minor GC 清新生代，较频繁；Full GC 常因老年代或元空间不足，停顿大，应通过堆大小、收集器选型和代码优化减少。

# 分布式锁

## 需求

在分布式/集群环境下，保证**互斥**、**高可用**、**可释放**（防死锁）。

| 方案 | 互斥 | 性能 | 说明 |
|------|------|------|------|
| MySQL | 唯一索引 / `for update` | 一般 | 依赖 DB，有单点风险 |
| Redis | `SET NX EX` | 高 | 最常用，需处理超时与原子释放 |
| ZooKeeper | 临时顺序节点 | 一般 | 强一致，实现复杂 |

## Redis 基础实现

### 加锁（推荐）

```bash
SET lock_key unique_value NX EX 30
```

- `NX`：不存在才设置（互斥）
- `EX 30`：30 秒过期（防死锁）
- `unique_value`：线程/请求唯一标识（如 UUID），**防止误删他人锁**

### 错误示例

```bash
SETNX lock_key 1        # 无过期 → 宕机死锁
DEL lock_key            # 不校验 value → 可能删别人锁
```

## 释放锁：Lua 保证原子性

> **已修正**：删除的 key 是 `KEYS[1]`，不是 `ARGV[1]`。

```lua
-- unlock.lua
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
```

```java
// KEYS[1] = lock_key, ARGV[1] = unique_value
redisTemplate.execute(script,
    Collections.singletonList(lockKey),
    uniqueValue);
```

## 常见问题与方案

| 问题 | 原因 | 方案 |
|------|------|------|
| 业务未完成锁过期 | 执行时间 > TTL | Redisson 看门狗续期；或合理 TTL + 业务幂等 |
| 误删他人锁 | 先 GET 再 DEL 非原子 | Lua 脚本 |
| 主从切换丢锁 | 主宕机锁未同步到从 | Redlock（有争议）；集群 + 业务容忍 |
| 不可重入 | 同线程无法再次加锁 | Redisson `RLock` |
| 不可重试 | 一次失败即返回 | 自旋 + 退避重试 |

## Redisson（生产推荐）

- 可重入锁、公平锁、读写锁
- **看门狗**：默认 30s，任务未完成自动续期
- 封装 Lua，避免手写错误

## 流程图

```
尝试 SET key value NX EX
    ├─ 失败 → 等待/重试/返回
    └─ 成功 → 执行业务
              ├─ 正常结束 → Lua 校验 value 后 DEL
              └─ 宕机 → 超时自动释放（需业务幂等）
```

## 面试一句话

> Redis 锁用 `SET NX EX` + 唯一 value；释放用 Lua 原子校验再删；生产推荐 Redisson 续期与可重入。
